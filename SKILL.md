# 业务分析与流程生成器

## 能力概述
本 Skill 通过六个渐进阶段，将现有业务文档体系转化为可复用的业务知识模板，再基于模板为新业务生成完整的流程与文档体系。适用于任何业务场景，不限于编程或技术项目。

**内置网络搜索能力**：在各阶段可实时通过网页搜索获取行业法规、标准模板、最佳实践、竞品分析、市场数据等外部信息，弥补内部文档的信息不足，让生成的模板和流程更贴合行业实际。

## When to Use

**触发条件：**
- 用户提供了包含业务文档的文件夹路径，需要分析文档内容
- 用户需要生成业务理解模板或知识库
- 用户需要为新业务设计完整流程并生成配套文档
- 用户提到"业务分析"、"流程生成"、"文档体系"等关键词

**症状识别：**
- "帮我分析这个文件夹里的业务文档"
- "基于现有制度，生成 XX 业务的流程"
- "我需要一套完整的 XX 管理文档"
- "把这些业务文档整理成模板"

**何时不使用：**
- 用户只需要单个文档的编辑或格式转换（使用文档编辑工具）
- 用户需要技术文档或代码文档（使用技术文档工具）
- 用户只需要简单的信息提取（使用信息提取工具）

## Quick Reference

| 阶段 | 目标 | 核心输出 | 网络搜索介入点 |
|------|------|----------|-----------------|
| Phase 1 | 扫描与采集 | 完整文档索引、内容摘要、元数据 | — |
| Phase 2 | 理解与关联 | 业务模块划分、文档关联图谱 | 搜索法规、术语、行业背景 |
| Phase 3 | 生成模板 | 业务理解模板（7 大要素） | 搜索行业标准、分类体系 |
| Phase 4 | 需求采集 | 自定义业务需求描述 | 搜索竞品、监管政策、最佳实践 |
| Phase 5 | 流程设计 | 新业务流程文档（含异常流程） | 搜索行业标准流程、法规要求 |
| Phase 6 | 文件生成 | 全套业务文件（匹配原格式） | 搜索合同范本、行业通用条款 |

## 网络搜索能力

本 Skill 内置网络搜索能力，在执行各阶段时可主动搜索互联网获取外部信息，弥补业务文档可能缺失的行业背景、法规要求、市场标准等。

### 搜索触发场景

| 场景 | 搜索内容 | 适用阶段 |
|------|----------|----------|
| 行业背景调查 | 行业政策、监管要求、市场趋势 | Phase 2, Phase 4 |
| 法规合规核查 | 相关法律法规、合规标准 | Phase 2, Phase 5 |
| 标准模板参考 | 行业标准文档模板、合同范本 | Phase 3, Phase 6 |
| 竞品/同行分析 | 竞品流程、行业最佳实践 | Phase 4 |
| 术语定义确认 | 专业术语、行业定义、分类标准 | Phase 2, Phase 3 |
| 流程基准对比 | 行业标杆流程、标准操作程序 | Phase 5 |
| 格式规范参考 | 行业文档格式标准、排版规范 | Phase 6 |

### 搜索执行规范

1. **搜索时机**：
   - 文档中发现不明确的行业术语、法规引用时 → 立即搜索
   - 业务领域信息不足以支撑模板生成时 → 搜索补充
   - 需要验证文档中的制度是否合规时 → 搜索确认
   - 设计新流程需要行业基准时 → 搜索参考

2. **搜索结果处理**：
   - 搜索到的外部信息必须标注来源（URL）
   - 与内部文档内容交叉验证（内部文档 > 外部信息 > 内部推测 > 默认值）
   - 如果搜索结果与文档内容冲突，标记差异点供用户确认
   - 将搜索结果整合到业务分析结论中，作为"行业参考"标注

3. **优先搜索源**：
   - 政府/监管机构官网（.gov 域名）
   - 行业协会标准文件
   - 知名咨询机构报告摘要
   - 行业头部企业公开文档
   - 专业百科/知识库

4. **禁止事项**：
   - 不得搜索用户内部机密信息（非公开的财务数据、客户信息等）
   - 不得将搜索到的信息直接作为"事实"而不加验证
   - 不得在未告知用户的情况下将文档内容对外搜索

## 执行阶段

### Phase 1: 文件夹扫描与文档采集
**目标**：完整获取目标文件夹中的所有业务文档内容与结构，支持所有文件格式

**重要原则**：本阶段必须完成所有文件的读取或处理（成功读取或记录元数据），确认无遗漏后才能进入 Phase 2。

1. **请求用户输入**：询问目标文件夹路径（如 `./公司制度`、`/path/to/业务流程文档`）

2. **扫描文档结构**：列出所有文件（排除临时文件、缓存等）
   ```bash
   find {folder_path} -type f -not -path '*/.DS_Store' -not -path '*/Thumbs.db' -not -path '*/~$*' -not -path '*/node_modules/*' -not -path '*/.git/*' -not -path '*/__pycache__/*' -not -path '*.swp'
   ```
   - 记录文件总数，向用户报告扫描结果
   - 如果文件数量过多（超过 500 个），与用户确认是否需要缩小范围或分批处理

3. **文件格式识别与分类**：
   - **办公文档**：.docx, .xlsx, .pptx, .doc, .xls, .ppt, .pdf, .odt, .ods, .odp
   - **网页与标记**：.html, .htm, .md, .markdown, .rst, .txt, .rtf
   - **数据与代码**：.csv, .tsv, .json, .xml, .yaml, .yml, .sql, .py, .js, .java, .go, .cpp, .log
   - **数据库与特殊**：.db, .sqlite, .sqlite3, .eml, .msg, .vcf, .ics
   - **压缩文件**：.zip, .rar, .7z, .tar, .gz, .bz2, .xz
   - **图片与多媒体**：.jpg, .jpeg, .png, .gif, .bmp, .svg, .tiff, .webp
   - **其他格式**：.css, .sh, .bat, .ps1, .ini, .conf, .cfg, .properties

4. **逐文件读取内容**：
   - **必须读取所有文件**，不允许跳过或抽样
   - **根据文件格式选择读取方法**（详见 `references/troubleshooting.md`）：
     
     **可直接读取的文本格式**：
     - .txt, .md, .csv, .json, .xml, .html, .css, .js, .py, .sql, .log, .sh, .ini 等
     - 方法：直接读取或简单解析
     
     **需要专用库的办公文档**：
     - .docx → python-docx
     - .xlsx → openpyxl
     - .pptx → python-pptx
     - .pdf → PyPDF2 或 pdfplumber（文本），pytesseract（扫描件）
     
     **旧版或特殊格式**：
     - .doc → antiword 或 LibreOffice 转换
     - .eml → email 库解析
     - .db → sqlite3 库连接
     - .zip → zipfile 模块解压后递归处理
     
      **图片文件**（需先判断是否包含业务文字信息）：
      - 优先通过文件名和所在目录判断图片用途（如截图、扫描件含文字，装饰图/照片可跳过）
      - 确认包含业务文字后 → pytesseract OCR 识别
      - 无法判断时，询问用户图片是否包含关键业务信息
      - .svg → 当作 XML 解析提取文本
   
   - 先扫描文件名清单，识别文件类型和业务归属，再按优先级顺序读取：
     - 第一批：目录索引、总览文档、核心制度文件
     - 第二批：具体流程文档、操作细则、管理办法
     - 第三批：配套表格、申请表单、统计台账
     - 第四批：参考材料、历史记录、辅助文档
   - 对于每个文件，记录：文件路径、文件类型、文档标题、核心内容摘要、读取方法

5. **无法读取文件的处理**（六层工具链自动流水线）：
   当遇到无法读取的文件时，**不允许直接跳过或标注"不可读"就放弃**。
   - **必须引用**：加载 `references/troubleshooting.md` 查看完整的故障排除流程
   - **核心流程**：按六层工具链逐级尝试，每层遍历所有可用工具，全部失败才进入下一层：
     1. **Level 1 — 专用解析库**：按格式匹配专用 Python 库（python-docx、openpyxl、PyMuPDF 等），缺失时自动 pip install
     2. **Level 2 — 通用转换器**：尝试跨格式工具（Docling、pandoc、MarkItDown）
     3. **Level 3 — CLI 工具集**：尝试系统级工具（libreoffice --cat、antiword、pdftotext 等）
     4. **Level 4 — OCR / 兜底**：图片文字识别（Tesseract、EasyOCR）+ 二进制 strings 提取
     5. **Level 5 — 手动接管**：用户粘贴/描述文件内容（A/B/C 三选项）
     6. **Level 6 — 元数据兜底**：记录文件元数据，基于文件名推断业务含义
   - **关键原则**：遍历所有层级和工具后才进入手动接管，避免过早放弃
   - **强制要求**：只有在所有文件都经过处理（成功读取、手动接管或记录元数据）后，才能进入 Phase 2
   - **格式特定处理**：针对特定格式的诊断方法和完整工具链见 `references/troubleshooting.md` 的详细分类

6. **建立完整文档索引**（纯编目性质，不做业务分析）：
   - 记录每个文档的路径、类型、页数/长度、文件格式、文件大小
   - 对未能读取内容的文件，记录其元数据、推断的业务含义和内容来源（自动读取/手动粘贴/用户描述/兜底推断）
   - 记录文件名间的表面关联（如同目录、文件名相似等），**注意：这只是文件名层面的观察，不代表业务关联分析——真正的业务关联分析在 Phase 2 完成**
   - **格式特征记录**：记录每种格式的文件数量、读取成功率、常用工具

---
**Phase 1 阶段结束 → 必须进入 Phase 2**

Phase 1 完成了文件采集和编目。**禁止跳过 Phase 2 直接进入 Phase 3 或后续阶段**。Phase 2 的"业务理解与关联分析"是 Phase 3 生成模板的必经前置步骤，缺失 Phase 2 的输出将导致模板内容空洞、脱离实际业务。

---

### Phase 2: 业务理解与关联分析
**目标**：从文档内容中提炼业务逻辑，梳理文档间的关联关系

**前置条件**：必须完成 Phase 1（文件夹扫描与文档采集），获得完整的文档索引后才能开始本阶段。

**重要原则**：本阶段是 Phase 3 生成模板的**唯一前置输入来源**。未完成本阶段的业务理解和关联分析，不得进入 Phase 3。本阶段的所有分析结果将直接作为 Phase 3 模板的素材。

1. **单文档分析**（必须全部完成，不允许跳过）：对所有文档逐个分析，其中制度类、流程类文档做详细分析，配套表格和模板做关联性简析：
   - 文档类型与用途（是制度、流程、还是表格？）
   - 适用的业务领域与场景
   - 涉及的角色/部门/岗位
   - 包含的业务规则与约束条件
   - 输入（需要什么前置条件/材料）
   - 输出（产生什么结果/文档）

2. **外部信息补充（网络搜索）**：
   分析文档时如遇到以下情况，立即进行网页搜索补充：
   - 文档中引用了未提供的法规名称 → 搜索该法规最新版本和核心条款
   - 出现了不熟悉的行业术语/标准 → 搜索术语定义和行业标准
   - 业务领域涉及监管合规要求 → 搜索最新监管政策和合规要求
   - 需要了解行业规模、市场数据 → 搜索行业报告和数据
   - 示例：文档提到"根据《个人信息保护法》"但未附条文 → 搜索并补充核心合规要求

3. **跨文档关联分析**：
   - 引用关系（A文档引用了B文档的哪条制度）
   - 上下游关系（申请表 → 审批表 → 执行单）
   - 依赖关系（流程文档依赖制度文档中的权限定义）
   - 互补关系（制度文档 + 操作细则 + 配套表格）

4. **业务模块划分**：将文档按业务领域聚类
   - 如：采购管理模块、费用报销模块、客户管理模块
   - 如：产品研发流程、上线发布流程、运维保障流程

5. **绘制业务关联图谱**（文本形式）：
   ```
    [采购申请制度] --引用--> [供应商管理办法]
    [费用报销流程] --依赖--> [财务审批权限表]
    [项目立项方案] --触发--> [资源分配流程]
    ```

---
**Phase 2 阶段结束 → 必须进入 Phase 3**

**禁止跳过 Phase 2 的任何步骤**。Phase 2 的业务理解和关联分析结果（单文档分析、跨文档关联、业务模块划分、关联图谱）是 Phase 3 生成模板的**唯一输入来源**。如果 Phase 2 未完成，Phase 3 无法生成有实际业务内容的模板。

---

### Phase 3: 生成业务理解模板
**目标**：输出标准化的业务理解文档，作为后续自定义业务的"知识基座"

**前置条件（必须全部满足）**：
1. Phase 1（文件夹扫描与文档采集）——必须已完成
2. Phase 2（业务理解与关联分析）——必须已完成，并获得以下输出：单文档分析结果、跨文档关联关系、业务模块划分、业务关联图谱
3. **如发现 Phase 2 未完成，立即停止 Phase 3，退回 Phase 2 补全后再继续**

**必须引用**：加载 `references/business-analysis-template.md` 作为输出格式规范

**网络搜索辅助**：生成模板前，搜索以下内容以增强模板质量：
- 行业通用的业务分类标准（如 GICS、行业分类标准）
- 同行业企业的组织架构和职责划分模式
- 行业法规对业务流程的强制性要求
- 典型的行业 KPI 和考核指标

输出内容应包含：
1. **业务全景概述**：一句话描述业务体系，一段话详细说明覆盖范围
2. **核心领域模型**：业务实体（如客户、订单、供应商）及其关系
3. **模块架构图**：各业务模块的职责边界与交互方式
4. **文档关联矩阵**：关键文档间的引用、依赖、互补关系表
5. **业务流程清单**：现有体系支持的所有核心流程（如采购流程、入职流程）
6. **业务规则与约束**：审批权限、金额门槛、时间限制、合规要求
7. **可复用模式**：可抽象为模板的通用业务解决方案
    - 如：三级审批模式、月度结算模式、供应商准入模式

---
**Phase 3 阶段结束 → 必须进入 Phase 4**

Phase 3 的业务理解模板已生成。**必须确保 Phase 1 和 Phase 2 已完成、Phase 3 模板已输出，才能进入 Phase 4**。Phase 4 的需求采集将基于 Phase 3 模板与用户对话。

---

### Phase 4: 自定义业务需求采集
**目标**：获取用户想要构建的新业务信息

**前置条件（必须全部满足）**：
1. Phase 3（生成业务理解模板）——必须已完成，模板已输出
2. 如果发现 Phase 3 未完成或有遗漏，退回补全后再继续

1. **展示模板摘要**：向用户展示 Phase 3 生成的业务理解模板核心要点（2-3段摘要）
2. **模板匹配判断**：评估用户描述的新业务与已有模板的匹配程度
   - **匹配度高的**（同领域，如模板是采购管理、新业务是供应商管理）→ 直接进入后续需求采集
   - **匹配度低的**（跨领域，如模板是采购管理、新业务是人力资源）→ 告知用户模板可能不适用，询问是否仍然基于模板调整，或跳过模板直接设计流程

3. **引导输入**：请用户描述想要构建的自定义业务，建议包含：
   - 业务名称与一句话描述
   - 核心功能需求
   - 与模板的相似点（可借鉴的部分）
   - 与模板的差异点（需要调整的部分）
   - 特殊约束（合规要求、行业特性、组织规模等）

3. **网络搜索辅助（需求阶段）**：在采集用户需求的同时，搜索外部信息辅助需求分析：
   - 搜索同行业类似业务的运作模式和流程
   - 搜索目标行业的最新监管政策和合规红线
   - 搜索行业头部企业的最佳实践和标准
   - 将搜索结果作为"行业参考"标注，帮助用户完善需求
   - 示例：用户说"要做供应链金融业务"→ 搜索供应链金融监管政策、主流业务模式、风控要求

4. **需求澄清**：如果用户输入模糊，通过反问澄清关键边界条件

5. **阶段回溯机制**：如果在后续 Phase 5/6 发现需求理解有偏差，允许用户回溯到 Phase 4 或 Phase 3 重新修订。技能应在各阶段起始处询问用户："此前的分析是否符合您的预期？需要调整吗？"

---
**Phase 4 阶段结束 → 必须进入 Phase 5**

**禁止跳过 Phase 4 直接进入 Phase 5**。Phase 4 采集的用户需求是 Phase 5 流程设计的唯一业务输入。未完成需求采集就开始设计流程，会导致流程与用户实际需求脱节。

---

### Phase 5: 生成全新业务流程
**目标**：基于模板知识，为新业务设计完整流程

**前置条件（必须全部满足）**：
1. Phase 3（业务理解模板）——必须已完成
2. Phase 4（自定义业务需求采集）——必须已完成，获得用户明确的新业务需求
3. **如发现 Phase 4 需求采集未完成或需求不明确，退回 Phase 4 补全后再继续**

**必须引用**：加载 `references/workflow-design-template.md` 作为输出格式规范

**网络搜索辅助（流程设计阶段）**：设计流程前和过程中搜索以下内容：
   - 行业标杆企业的同类流程结构和步骤
   - 行业法规对流程环节的强制性要求（如必须有哪些审核节点）
   - 行业标准流程框架（如 ISO 标准流程、ITIL 流程框架等）
   - 常见异常场景和处理方式（行业通行的异常处理模式）
   - 将搜索到的行业标准标注为"行业参考"，与模板模式交叉验证

设计步骤：
1. **模式映射**：将模板中的业务模式映射到新业务场景
   - 模板中的"三级审批模式" → 新业务中的"XX审批模式"
   - 模板中的"月度结算流程" → 新业务中的"XX结算流程"

2. **流程设计**：
   - 核心业务流程（主成功场景，含步骤、角色、产出物）
   - 异常处理流程（驳回、退回、超时、特殊情况）
   - 并行与串行处理（哪些步骤可并行，哪些必须串行）
   - 跨部门协作流程（需要哪些部门/岗位配合）

3. **业务规则设计**：基于新业务的领域模型，设计规则与约束
   - 搜索确认行业通用的审批权限分层标准
   - 搜索确认行业法定的时限要求

4. **文档流转设计**：定义流程中各环节的输入文档、输出文档、存档要求，其中"所需文档清单"将直接作为 Phase 6 的文件生成输入

5. **输出流程文档**：包含流程描述、角色职责、文档清单、时效要求

---
**Phase 5 阶段结束 → 必须进入 Phase 6**

**禁止跳过 Phase 5 直接进入 Phase 6**。Phase 5 输出的文档清单和流程步骤是 Phase 6 生成业务文件的直接依据。如果 Phase 5 未完成，Phase 6 无法确定需要生成哪些文件及其内容。

---

### Phase 6: 补充流程所需业务文件（匹配目标文件夹中的文档格式）
**目标**：生成新业务所需的全部业务文件，**文件格式必须严格匹配目标文件夹中的对应文档格式**

**前置条件（必须全部满足）**：
1. Phase 5（业务流程设计）——必须已完成，获得完整的流程文档和所需文档清单
2. Phase 1 中目标文件夹的文档格式分析——必须已完成，作为格式匹配依据
3. **如发现 Phase 5 未完成或文档清单缺失，退回 Phase 5 补全后再继续**

**必须引用**：加载 `references/file-structure-template.md` 作为输出格式规范

> **重要说明**：这里的"目标文件夹"就是 Phase 1 中用户指定的那个文件夹。本阶段生成的新业务文件，格式必须和目标文件夹里的业务文档保持一致。

**核心规则**：
1. **格式继承**：生成的新文件内容结构必须与目标文件夹中的同类业务文档完全一致
   - 如果目标文件夹里有 Word 文档（.docx）→ 新文件输出 Word 格式的内容框架
   - 如果目标文件夹里有 Excel 表格（.xlsx）→ 新文件输出表格格式的内容框架
   - 如果目标文件夹里有 PPT（.pptx）→ 新文件输出 PPT 格式的内容框架
   - 如果目标文件夹里有 Markdown（.md）→ 新文件也用 Markdown
   - 如果目标文件夹里有 PDF → 新文件生成可转为 PDF 的内容
   - 以此类推

   > **重要说明**：LLM 无法直接生成 .docx/.xlsx/.pptx 等二进制文件。对于这些格式，输出内容结构（章节、表格、字段等）与目标文件完全一致，并提供 Python 脚本（使用 python-docx/openpyxl/python-pptx）将内容转换为实际二进制文件。用户可按脚本生成最终文件。

2. **结构继承**：文档的章节结构、标题层级、内容组织方式必须与目标文件夹中的文档保持一致

3. **风格继承**：语言风格（正式/口语化）、术语体系、编号方式必须与目标文件夹中的文档保持一致

4. **模板继承**：表格样式、审批单格式、通知模板等必须与目标文件夹中的模板保持一致

执行步骤：
1. **分析目标文件夹中的文档格式**：
   - 记录每种文档类型的扩展名、排版特征、结构特征
   - 提取文档风格特征（语言风格、术语体系、编号方式、审批层级）
   - 记录配套表格/表单的格式（字段、布局、必填项标记方式）

2. **网络搜索辅助（文件生成阶段）**：生成前搜索以下内容增强文件专业性：
   - 行业合同/协议的标准条款和必备要素
   - 行业申请表的通用字段和布局
   - 行业报告/方案的目录结构和章节规范
   - 行业合规文件必须包含的声明条款
   - 注意：生成的文件格式仍需严格匹配目标文件夹中的文档格式，搜索到的标准仅用于补充内容

3. **文件清单生成**：根据业务流程，列出所需的所有业务文件（按模块组织），格式与目标文件夹中的文档匹配
   ```
   新业务文档/
   ├── 制度类/
   │   ├── XX管理办法.docx      # 格式同目标文件夹中的"供应商管理办法.docx"
   │   └── XX实施细则.docx      # 格式同目标文件夹中的"费用报销实施细则.docx"
   ├── 流程类/
   │   ├── XX操作流程.md        # 格式同目标文件夹中的"采购操作流程.md"
   │   └── XX审批流程图.pptx    # 格式同目标文件夹中的"审批流程图.pptx"
   ├── 表格类/
   │   ├── XX申请表.xlsx        # 格式同目标文件夹中的"采购申请表.xlsx"
   │   └── XX统计台账.xlsx      # 格式同目标文件夹中的"月度统计台账.xlsx"
   └── 模板类/
       ├── XX通知模板.docx      # 格式同目标文件夹中的"会议通知模板.docx"
       └── XX合同模板.docx      # 格式同目标文件夹中的"服务合同模板.docx"
   ```

4. **文件内容生成（按目标文件夹中的文档格式）**：
   - 对每个文件，生成**与目标文件夹中的同类文档同格式**的完整内容框架
   - 使用目标文件夹中的文档风格（如正式公文风格、操作手册风格）
   - 使用目标文件夹中的术语体系和编号方式
   - 包含必要的章节、条款、表格、审批栏
   - 添加 `[待填写]` 标记需要用户补充的具体业务内容
   - 确保文档间的引用关系正确

   **示例映射**：
   | 目标文件夹中的文档 | 原始格式特征 | 新文件格式 |
   |-------------------|-------------|-----------|
   | `供应商管理办法.docx` | 正式公文、章节式、含审批权限表 | `渠道商管理办法.docx` - 同格式 |
   | `费用报销流程.md` | Markdown、步骤式、含角色职责 | `差旅报销流程.md` - 同格式 |
   | `采购申请表.xlsx` | Excel、字段式、含下拉选项 | `设备采购申请表.xlsx` - 同格式 |
   | `项目立项PPT.pptx` | PPT、分页式、含流程图占位 | `产品上线方案.pptx` - 同格式 |
   | `会议纪要模板.docx` | Word、表格式、含签到栏 | `周会纪要模板.docx` - 同格式 |

5. **配套文档生成**：
   - 制度文档配套的细则、解释说明，格式与目标文件夹中的配套文档一致
   - 流程文档配套的申请表、审批单，格式与目标文件夹中的表单一致
   - 方案文档配套的预算表、排期表，格式与目标文件夹中的表格一致

5. **与 Phase 5 的文档清单对接**：直接使用 Phase 5 输出的"所需文档清单"作为本步骤的文件生成依据，确保不遗漏

6. **一致性检查**：
   - 检查所有文档间的引用是否准确
   - 检查流程步骤与配套表格的字段是否匹配
   - 检查审批权限与制度定义是否一致
   - 检查文件格式与目标文件夹中的同类文档一致

7. **输出文件**：将生成的业务文件内容输出，供用户保存到指定目录

8. **阶段间回溯**：输出前询问用户是否需要对前面的分析（Phase 3-5）进行调整，避免到最后才发现偏差

## 输出规范
- Phase 1-5 的输出使用 Markdown 格式，便于阅读和后续编辑
- **Phase 6 生成的实际业务文件内容必须匹配目标文件夹中的文档格式**，不是 Markdown
- 在阶段转换时，明确告知用户当前进度和下一步行动
- 生成的文档内容应可直接用于实际业务场景

## 边界与约束
- 文件数量过多时（超过 500 个），与用户确认是否需要缩小范围或分批处理，但当前批次内必须全部读取完成
- 不执行任何删除、修改原始文件夹的操作，只读取
- 生成文件前需确认输出目录，避免覆盖现有文件
- 对于无法读取的文件，触发六层工具链流水线（Level 1-6），每层遍历所有可用工具，全部失败才进入下一层，不可无限循环
- 如果目标文件夹包含多种文档格式，优先使用占比最高的格式
- **网络搜索约束**：
  - 搜索仅用于补充内部文档信息不足，不得替代对用户文档的完整读取和分析
  - 搜索到的外部信息与用户文档内容冲突时，标注差异并请用户确认
  - 避免过度搜索——每个阶段最多执行 3-5 次关键搜索，聚焦核心信息缺口
  - 搜索结果必须可追溯（记录来源 URL），不得使用未标记来源的外部信息

## Common Mistakes

**错误 1：未完成前置阶段就跳入后续阶段**
- **症状**：未读取完所有文件就进入 Phase 2；或 Phase 1 完成后直接跳到 Phase 3，跳过了 Phase 2 的业务理解与关联分析
- **后果**：生成的模板只有文件清单和表面信息，缺少业务逻辑和关联关系
- **修复**：严格遵循 Phase 1 → Phase 2 → Phase 3 → ... → Phase 6 的顺序，**每个阶段完成后将输出传递给下一阶段，禁止跨阶段跳跃**

**错误 2：Phase 6 生成文件格式不匹配目标文件夹**
- **症状**：目标文件夹是 .docx，但生成了 Markdown 文件
- **后果**：用户无法直接使用生成的文件，需要手动转换
- **修复**：Phase 6 必须先分析目标文件夹的文档格式特征，再生成对应格式

**错误 3：未处理无法读取的文件就跳过**
- **症状**：遇到无法读取的文件，直接标注"不可读"后跳过
- **后果**：丢失业务信息，影响后续分析准确性
- **修复**：必须执行完整的六层工具链流水线（Level 1-4 自动尝试 → Level 5 手动接管 → Level 6 兜底处理），详见 references/troubleshooting.md

**错误 4：Phase 1 完成后跳过 Phase 2，直接进入 Phase 3 生成模板**
- **症状**：扫描完文件后直接开始生成业务理解模板，跳过了业务理解和关联分析
- **后果**：模板只有文件清单和表面信息，缺失业务逻辑、角色职责、业务流程间的关联关系，导致后续 Phase 5 无法基于模板设计出可用的流程
- **修复**：Phase 1 完成后**必须执行 Phase 2**（业务理解与关联分析），获得单文档分析、跨文档关联、业务模块划分和关联图谱后，才能进入 Phase 3

**错误 5：Phase 3 模板生成过于抽象，无法复用**
- **症状**：模板只包含通用描述，没有具体的业务规则和模式
- **后果**：Phase 5 无法基于模板生成具体业务流程
- **修复**：模板必须包含可复用的业务模式（如"三级审批模式"）和具体规则（如"金额>5000 需总监审批"）

**错误 6：未向用户确认就执行批量操作**
- **症状**：文件数量超过 500 个时，自动决定分批处理
- **后果**：可能违反用户意图，遗漏重要文件
- **修复**：文件数量超过 500 时，必须向用户确认处理范围

**错误 7：文档信息不足时未使用网络搜索补充**
- **症状**：文档中引用了外部法规/标准但未提供原文，直接跳过或凭猜测处理
- **后果**：生成的模板和流程可能违反实际法规要求，或与行业标准脱节
- **修复**：遇到不明确的法规引用、行业术语、标准要求时，先搜索确认再继续

**错误 8：搜索结果未标记来源直接使用**
- **症状**：搜索到的行业数据或法规条文未记录 URL，直接作为业务规则写入
- **后果**：用户无法追溯信息来源，无法验证信息准确性
- **修复**：每个搜索结果的引用必须标注来源 URL，必要时应请用户复核
